# DayCost 演进与优化蓝图 V2.0 (Production-Ready Blueprint)

在我们前期的迭代中，DayCost 已经从一个基础原型蜕变成了一个具备**高级视觉质感、虚拟长列表滚动、多维图表统计、PWA 离线支持、RBAC 权限管理以及分页加载**的“工业级小钢炮”。

## 一、 核心功能与玩法创新 (Functional Innovations)

除了现有的基础记账，DayCost 可以向“智能资产管家”方向进化：

### 1. 标签与多维分类系统 (Tagging System)
- **痛点**：目前物品只有“父子包含”关系。如果想知道“我为了玩游戏到底花了多少钱”或者“摄影爱好的日均成本是多少”，跨设备的统计很难实现。
- **优化**：引入 `#游戏`、`#生产力`、`#冲动消费` 等自定义标签系统。在统计面板中增加“按标签聚合”的饼图，让每一分钱的去向变得立体。

### 2. 真实折旧曲线与残值预估 (Depreciation Models)
- **痛点**：目前 DayCost 使用的是简单的“直线折旧法”（总价 / 天数）。但在真实世界中，电子产品第一年贬值最快，之后逐渐平缓。
- **优化**：加入不同的折旧算法（如：直线折旧、双倍余额递减法）。甚至可以根据用户购买时间，在面板中实时预估该物品“今天挂闲鱼还能卖多少钱（当前残值）”。

### 3. 附件图片与电子小票 (Image & Receipt Attachments)
- **痛点**：纯文字列表对于实体物品来说略显枯燥，且时间久了容易忘记保修凭证在哪。
- **优化**：允许在物品卡片上长按上传“实物图”或“发票/保修单截图”。配合瀑布流布局，将 DayCost 变成一个赏心悦目的个人数字资产陈列室。

### 4. 批量操作与级联退役 (Batch Operations)
- **痛点**：如果你把一台组装电脑（包含主板、CPU、显卡等10个子配件）整体打包卖掉了，现在需要挨个点开子配件去标记为“已售出”。
- **优化**：实现级联状态变更。当标记一个父级目录“已回血/已损坏”时，弹窗询问是否一键将名下所有子配件同步打包处理，极大地降低维护成本。

### 5. 预算预警与消费节流 (Budgeting & Alerts)
- **优化**：设定一个心理预期的“日均耗散警戒线”（例如：每天所有设备消耗不超过 50 元）。当新购入物品导致日均线爆表时，界面呈现视觉警告，从而真正起到“遏制消费冲动”的核心产品目的。

## 二、 核心安全与防护架构 (Security & Protection)

作为涉及金钱记录的系统，一旦对外公开，安全将是第一位的。

### 1. 流量防刷与限流 (Rate Limiting)
- **痛点**：目前后端的 API 完全裸奔。如果有人用脚本疯狂注册账号或者请求数据，你的服务器内存会被瞬间打满，甚至数据库被锁死。
- **优化**：引入 `express-rate-limit`，限制同一个 IP 一分钟内最多请求 100 次，并且对注册/登录接口施加更严厉的限制（如每小时最多错 5 次密码）。

### 2. HTTPS 与证书配置 (SSL/TLS)
- **痛点**：PWA 功能和用户的密码传输必须在加密通道下进行，否则很容易被中间人拦截嗅探。
- **优化**：在正式服务器上必须配置 Nginx 反向代理，并使用 Let's Encrypt 配置免费的 HTTPS 证书，强制所有 HTTP 请求跳转至 HTTPS。

### 3. 数据校验与防注入 (Data Sanitization)
- **痛点**：目前前端传过来的名字和金额，虽然做了一点基础判断，但没有进行严格的 XSS 清洗。
- **优化**：在 Node.js 中引入验证库（如 `express-validator` 或 `Joi`），对用户输入的每一句话进行 HTML 标签剥离和转义，防止恶意脚本注入。

## 二、 工业级部署与运维 (Deployment & Infrastructure)

### 1. 数据库升维 (Database Migration)
- **痛点**：当前使用的是单文件数据库 `SQLite`。这对于个人或者小团队堪称完美，但如果有几百上千人同时高频读写，SQLite 会因为文件锁问题产生严重的并发瓶颈。
- **优化**：利用 ORM 工具（如 Prisma 或 Sequelize）将底层数据库平滑迁移到 `PostgreSQL` 或 `MySQL`，彻底解放并发能力。

### 2. 容器化部署 (Dockerization)
- **痛点**：换一台服务器需要重新装 Node、配环境、拷贝代码，极其繁琐且容易出错。
- **优化**：编写 `Dockerfile` 和 `docker-compose.yml`，将 Node 后端、数据库、Nginx 全部打包隔离。以后部署或迁移，只需要一行命令 `docker-compose up -d` 即可。

### 3. 自动化数据备份 (Automated Backups)
- **优化**：设置服务器定时任务（Cron），每天凌晨自动将数据库快照导出并打包上传到云存储（如阿里云 OSS 或 AWS S3），即使服务器物理损坏也能找回用户资产。

---

## 三、 用户闭环与商业化 (Business Features)

### 1. 密码找回与身份验证 (Email/SMS Verification)
- **痛点**：现在用户忘了密码只能抓瞎，或者求助你这位“超级管理员”去数据库帮他改。
- **优化**：接入邮件发送服务（如 NodeMailer 配合 SMTP），实现“找回密码”、“注册验证”的闭环。

### 2. 用户数据私有权 (Data Export)
- **痛点**：对于这种记录型软件，用户最害怕“被平台绑架”。
- **优化**：增加一键“导出为 Excel (CSV) 或 JSON”功能。大大增加产品在用户心中的信任度和专业感。

### 3. 主题自由度 (Theming)
- **优化**：目前 DayCost 是强制极其酷炫的极夜/暗黑（Dark）风格。对于公开网站，应当支持检测用户的系统偏好（`prefers-color-scheme`），并提供一套优雅明亮的白昼主题。

---

**总结：**
DayCost 在 **代码逻辑、视觉 UI 和单兵作战能力**上已经远超许多业余项目，现在距离“合格的商业网站”，主要缺的是**“外层装甲”**（Nginx 部署、安全限流、邮件闭环和容器化架构）。
