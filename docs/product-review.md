# DayCost 产品审视报告

> 审视人：许清楚（Xu）· 产品经理
> 日期：2026-06-26
> 版本：v1.0.0

## 概述

DayCost 以"每日摊销成本"为核心洞察，揭示物品持有期的隐性代价——"买得起，不一定用得起；放着吃灰，才是最昂贵的代价"。这一理念极具传播力与差异化价值。当前 v1.0.0 已实现核心账本、统计、TOTP 密钥管理、回收站、设置及管理后台六大模块，功能骨架完整。但在用户体验深度、功能闭环、技术健壮性、安全合规、商业化路径及开发者体验方面仍有显著短板。本报告从 7 个维度逐一拆解，并给出可执行的优先级路线图。

## 1. 用户体验 (UX) Gap Analysis

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **无新手引导/Onboarding** | 新用户首次进入面对空白账本，无任何提示说明"如何理解日摊成本"这一核心概念，也无引导创建第一条记录。 | 添加 3 步 Onboarding：①解释日摊成本概念（图文/动画）；②引导添加第一件物品；③提示探索统计视图 |
| 2 | **空状态设计缺失** | 账本/统计/密钥各 Tab 在无数据时无视觉引导，用户看到空白表格或空图表即产生"产品无用"认知。 | 每个空状态配专属插画 + 一句话引导 + CTA 按钮（"添加第一件物品"） |
| 3 | **移动端手势冲突** | 长按触发批量管理、双击触发行内编辑，两者在同一交互对象上存在触发歧义；横向滑动手势可能与系统返回冲突。 | 明确手势层级：单击→查看详情，长按→批量选择，双击仅在桌面端生效；移动端用独立按钮代替双击 |
| 4 | **SPA 导航"更多"下拉可发现性差** | 回收站/密钥/设置藏在"更多"下拉里，新用户难以发现 TOTP 等特色功能，降低留存第一印象。 | 初版把密钥和回收站提升为独立 Tab Icon（共 5 Tab），设置保留在右上角齿轮图标；或用底部导航 + 右上角设置双入口 |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 5 | **统计视图缺少横向对比** | 趋势图仅展示时间维度，无法横向对比不同物品/标签的日摊成本排行，"哪个东西最亏"这一核心问题仍需心算。 | 增加 "日成本排行榜" 视图（按 ¥/day 降序），配合颜色编码（红=高成本、绿=低成本） |
| 6 | **树组合编辑体验割裂** | 创建父子关系需要在不同记录间手动设定 parent_id，缺乏可视化拖拽或树形编辑器，组合成本无法直观预览。 | 提供 "组合编辑器"：拖拽子项到父项下，实时显示组合日成本；支持树形缩进展示 |
| 7 | **状态切换无确认提示** | 物品标记为 broken/sold 后日摊计算逻辑立即变更，但无确认弹窗，误操作风险高且难以回退。 | 状态变更弹出确认对话框，明确提示 "日摊成本将重新计算" 并给出变更前/后对比预览 |
| 8 | **无键盘快捷键** | 桌面端高频操作（添加记录、切换 Tab、搜索）全靠鼠标/触屏，效率低。 | 添加常用快捷键：N=新建、Tab=切换视图、Ctrl+F=搜索、Esc=关闭弹窗 |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 9 | **搜索/筛选能力不足** | 无全局搜索框，无法按物品名、标签、价格区间快速定位记录。 | 添加顶部搜索栏 + 筛选器（标签、状态、日期范围、价格区间） |
| 10 | **无分享/截图功能** | 统计图表无导出图片或社交分享能力，"我的GPU日成本竟高达¥28"这类传播点被扼杀。 | 图表增加 "分享卡片" 功能：生成带品牌水印的可分享图片，适配社交媒体尺寸 |

## 2. 功能完整性 (Feature Completeness)

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **数据导入仅 CSV，无其他格式** | 只支持 CSV 导出/导入，不支持常见记账软件（如随手记、MoneyWiz）的数据迁移，用户换工具成本极高。 | 支持导入主流格式：CSV 增强（自动检测编码/分隔符）、JSON（DayCost 专属）、OFX/QIF（国际标准） |
| 2 | **TOTP 密钥管理缺少验证码实时刷新** | 有 QR 扫描和分组管理，但缺少动态 6 位码实时显示与倒计时进度条——这是 TOTP 管理器的核心价值。 | 增加 TOTP 代码实时生成面板：30s 倒计时环形进度条 + 一键复制，支持多密钥同时展示 |
| 3 | **回收站恢复后关联数据断裂** | 物品删除进入回收站后，其作为父项的子项关系、标签绑定、统计聚合均中断，恢复后需手动重建。 | 删除时软冻结关联关系（标记而非切断），恢复时自动重建 parent-child 和标签映射 |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 4 | **缺少货币/多币种支持** | 所有金额默认单一货币符号，无法处理跨境购买（如海外代购 GPU 用 USD 结算，日成本换算为 CNY）。 | 记录增加 currency 字段，统计视图增加汇率换算（可手动设汇率或接入免费 API） |
| 5 | **折旧计算缺少可视化对比** | 直线法与双倍余额递减法两种折旧方式的选择缺乏直观对比——用户不知道哪种更贴合自己的使用场景。 | 折旧方式选择时展示对比图表：两种方法的日成本曲线叠加，帮助用户直观决策 |
| 6 | **管理后台 Phase 3/4 未完成** | Analytics 和 System Settings 两个阶段尚未实现，管理员无法看到全局使用趋势、无法配置系统级参数。 | Phase 3：全局统计面板（活跃用户数、记录总量、日均新增）；Phase 4：系统参数（注册开关、维护公告、数据保留策略） |
| 7 | **无通知/提醒机制** | 物品闲置超阈值（如 90 天无状态变更）无提醒，"放着吃灰"这一核心理念缺乏主动触发。 | 增加 "闲置提醒"：物品 90 天无交互时推送通知 "你的 XXX 已闲置 ¥XX/天，考虑出售？"；支持自定义阈值 |
| 8 | **缺少预算/目标设定** | 有日成本统计但无对比基准，用户不知道 "¥15/天算高还是低"。 | 增加日成本预算目标：用户设月度持有成本上限，统计页实时显示预算消耗进度条 |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 9 | **无标签体系管理** | 标签目前作为记录字段存储，但无标签 CRUD 界面、无标签层级、无标签统计汇总。 | 增加标签管理页：创建/编辑/删除标签、标签颜色/图标、标签使用频次排行 |
| 10 | **社交/社区元素为零** | 单人工具无社区氛围，"哪个数码产品日成本最高"这类话题天然适合社区讨论。 | 可选：增加 "日成本排行榜" 匿名社区功能，用户自愿提交物品类别+日成本，生成热门品类排行 |

## 3. 技术债务与性能 (Technical Debt & Performance)

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **script.js 2347 行单体巨石** | 所有前端逻辑（路由、渲染、事件绑定、数据请求、图表）集中在一个文件，无法模块化测试、增量部署、团队协作。 | 拆分为模块架构：按功能域拆分（records-view.js, stats-view.js, totp-view.js, common-ui.js, api-client.js），用 ES Module 或轻量 bundler（esbuild）组织 |
| 2 | **零测试覆盖** | 无单元测试、无集成测试、无 E2E 测试，任何改动都可能引入回归，且无法验证折旧计算等核心算法的正确性。 | 最低优先：核心算法单元测试（折旧计算、日摊成本、树组合聚合）→ API 集成测试（auth、records CRUD）→ 关键路径 E2E（Cypress/Playwright） |
| 3 | **无 CI/CD 管线** | 无自动化构建、测试、部署流程，代码质量靠人工检查，发布靠手动操作。 | 引入 GitHub Actions：lint → test → build → deploy（Docker push），PR 门禁要求 test pass |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 4 | **Google Fonts 离线不可用** | PWA 支持离线使用但依赖 Google Fonts CDN，离线时字体回退导致排版错乱。 | 将字体文件本地化（放入 public/fonts/），或改用系统字体栈（Noto Sans SC → system-ui fallback） |
| 5 | **Chart.js/Clusterize.js 全量加载** | 无 tree-shaking 或按需加载，首屏加载冗余代码；Chart.js 全包 ~200KB，实际只用折线/饼图。 | Chart.js 按需 import（仅注册 LineController/DoughnutController）；Clusterify.js 考虑替换为原生虚拟滚动（Intersection Observer） |
| 6 | **SQLite 单写锁瓶颈** | WAL 模式改善并发读，但 SQLite 单写锁在高频写入场景（多用户同时操作）下仍可能阻塞。 | 短期：确保写入操作异步排队、事务尽量短；中长期：评估 PostgreSQL 迁移路径（保留 SQLite 作为单机版选项） |
| 7 | **版本缓存 ?v=N 查询字符串** | 资源版本化用 query string 方式，CDN/代理缓存命中率低，且手动维护版本号。 | 改用文件名哈希（contenthash），构建步骤自动生成；或至少用 Service Worker 缓存策略替代 |
| 8 | **style.css 3400 行 + admin.css 1700 行** | 样式文件同样臃肿，无 CSS 模块化，主题切换靠全局变量覆盖但无设计系统强制执行。 | 已有 design-system.css，应扩展为完整 Token 层 → Component 层 → Page 层的三层架构；按视图拆分 CSS 文件 |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 9 | **无前端状态管理规范** | state-manager.js 存在但文档中无使用说明，事件驱动混乱（直接 DOM 操作 + 事件 + 状态三层并存）。 | 统一为状态驱动渲染：state-manager 作为单一数据源，DOM 更新通过订阅机制触发，消除手动 DOM 操作 |
| 10 | **Docker 镜像未优化** | 未使用多阶段构建、Alpine 基础镜像，镜像体积偏大。 | 多阶段构建：build 阶段安装依赖 → runtime 阶段仅拷贝产物；基础镜像改 node:alpine |

## 4. 安全与合规 (Security & Compliance)

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **JWT_SECRET 硬编码默认值** | `daycost_dev_secret_key_999` 虽仅在 dev 模式使用且 prod 会抛错，但源码暴露默认值本身是风险——开发者可能在 prod 环境意外遗留。 | 移除源码中的默认值，改为启动时强制从环境变量读取；缺少时直接拒绝启动（不分 dev/prod） |
| 2 | **无 CSRF 防护** | SPA 使用 JWT 放在 localStorage，但 POST 请求无 CSRF Token，若存在 XSS 漏洞则可被跨站请求伪造攻击。 | 改 JWT 存储为 HttpOnly Cookie + SameSite=Strict；或至少对状态变更请求添加 CSRF Token（Double Submit Cookie 模式） |
| 3 | **密码无复杂度要求** | 仅校验 "不为空"，无最小长度、无大小写/数字/特殊字符要求，弱密码如 "1" 即可通过。 | 密码策略：最小 8 字符 + 至含 1 数字 + 1 大写字母；密码强度指示器；bcrypt rounds 提升至 12（当前默认 10） |
| 4 | **TOTP 密钥加密密钥管理薄弱** | TOTP_KEY 在 dev 环境随机生成并 warn，但随机值未持久化——每次重启生成新密钥则所有已存密钥解密失败。 | Dev 模式也需持久化 TOTP_KEY（写入 .env.local 或专用密钥文件）；或 dev 模式使用固定测试密钥（明确标注仅用于开发） |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 5 | **Genesis Admin 风险** | 第一个注册用户自动成为管理员，无审批/验证流程。若公开部署，任何最先访问者即获管理员权限。 | 首次部署需通过环境变量或 CLI 命令预设管理员账号；或 Genesis Admin 创建后需邮箱验证确认 |
| 6 | **Rate Limiting 间隙** | 存在 rate-limit 中间件，但需逐路由检查是否全覆盖——遗漏的路由（如 TOTP、Admin API）可能被暴力攻击。 | 全局 rate-limit 兜底（所有路由默认 100 req/15min），敏感路由（auth、admin）单独收紧（20 req/15min） |
| 7 | **备份文件无加密** | .daycost JSON 备份导出为明文，含所有财务数据与 TOTP 密钥，泄露风险极高。 | 导出时可选密码加密（AES-256-GCM），恢复时需输入密码解密；至少对 TOTP 密钥段单独加密 |
| 8 | **XSS 清洗不完整** | xssClean 中间件仅清洗请求体，但前端渲染仍存在 innerHTML 直接写入风险（script.js 中多处）。 | 前端统一使用 textContent/setAttribute 替代 innerHTML；对必须渲染 HTML 的场景使用 DOMPurify |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 9 | **无 GDPR/隐私合规声明** | 欧盟用户数据处理（邮箱、个人财务数据）缺乏隐私政策、数据删除权、Cookie 同意机制。 | 添加隐私政策页面；注册流程增加同意勾选；提供 "删除我的所有数据" 功能（GDPR Right to Erasure） |
| 10 | **审计日志不可导出** | audit_logs 表存在但无管理界面导出功能，安全事件追溯困难。 | Admin Phase 4 中增加审计日志查看与导出（CSV/JSON），支持按用户/时间/操作类型筛选 |

## 5. 商业化与增长 (Commercialization & Growth)

### 市场定位

DayCost 的核心差异化在于 **"日摊成本"而非"支出流水"**——这是对传统记账软件范式的颠覆。现有记账工具（随手记、MoneyWiz、薄荷记账）关注"我花了多少"，DayCost 回答"我持有多少代价"——后者更能触发消费反思与行为改变。

**定位建议**：个人资产成本意识工具（Asset Cost Awareness），而非传统记账 App（Expense Tracker）。

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **核心概念传达不足** | "日摊成本"理念在产品内几乎无教育路径，新用户无法快速理解价值主张，转化和留存天然低。 | Onboarding 强概念灌输 + 统计页核心洞察卡片（"你本月持有成本 ¥XXX，最昂贵物品：XXX ¥XX/天"） |
| 2 | **无病毒传播机制** | 产品天然适合"晒成本"社交传播（"我的 MacBook 日成本 ¥15"），但无分享功能，自增长为零。 | 生成"日成本名片卡"：物品名 + ¥/天 + 品牌水印，一键分享到微信/微博/X |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 3 | **留存驱动单一** | 用户仅在购买新物品时打开 App，日常使用频率低（不像流水记账有日常触发）。 | "闲置提醒"推送 + 月度持有成本报告邮件 + "本周最亏物品"推送——把被动查询变为主动触达 |
| 4 | **无国际化（i18n）** | 界面全中文，无多语言框架，海外市场（尤其日本/韩国的消费反思文化契合）无法触达。 | 引入 i18n 框架（至少中/英/日三语），优先翻译核心术语（日摊成本 = Daily Amortized Cost） |
| 5 | **无变现路径设计** | 无付费功能、无增值服务、无数据服务，纯公益模式无法支撑长期迭代。 | 分层策略：Free（基础账本+统计）→ Pro（多币种+预算+提醒+高级折旧+导出加密）→ Team（家庭共享账本） |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 6 | **无用户反馈渠道** | 产品内无反馈入口、无社区、无问卷，无法收集需求方向验证。 | 设置页增加 "反馈与建议"入口（GitHub Issues 链接或内置表单）；每月推送 1 次 NPS 调查 |
| 7 | **无数据驱动增长指标** | 无留存分析、无转化漏斗、无使用频次统计（自身不跟踪自身）。 | Admin Analytics 增加增长仪表盘：DAU/MAU、7日留存、核心功能使用率、记录创建频次分布 |

## 6. 文档与开发者体验 (Documentation & DX)

### 🔴 Critical

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 1 | **无 API 文档** | 6 个路由文件暴露的 REST API 无任何文档（无 Swagger/OpenAPI、无 README API 段、无注释规范）。 | 生成 OpenAPI 3.0 spec（可从 Express 路由自动提取），部署 Swagger UI 到 /docs 路径 |

### 🟡 Important

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 3 | **package.json 元数据缺失** | description 为空、无 keywords、无 test script、无 repository 字段——npm 生态和 GitHub 发现性极差。 | 补全：description、keywords（["expense","amortized-cost","asset-tracking","totp"]）、repository、scripts.test、scripts.lint |
| 4 | **无 ESLint/Prettier/Husky** | 代码格式和风格无自动化校验，PR 代码风格不一致，手动 review 效率低。 | 引入 ESLint + Prettier + Husky（pre-commit hook：lint-staged），统一代码风格基线 |
| 5 | **无环境变量文档** | JWT_SECRET、TOTP_KEY、CORS_ORIGIN 等关键配置无 .env.example 文件，部署者需读源码猜配置。 | 创建 .env.example + 部署文档段落，列出所有必需/可选环境变量及说明 |
| 6 | **docs/ 内文档碎片化** | 存在 ux-architecture-redesign.md、code-review-standards.md、admin-page-redesign.md、data-flow-design.md 等多个设计文档，但无索引和关联说明。 | 创建 docs/INDEX.md 作为文档导航入口，按类型分类（设计规范/安全审计/产品审视）并标注状态（已完成/进行中/待更新） |

### 🟢 Nice-to-have

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| 7 | **无 Changelog** | 版本迭代历史无记录，用户和贡献者无法了解每个版本的变化。 | 引入 CHANGELOG.md（遵循 Keep a Changelog 格式），或由 CI 自动从 git log 生成 |
| 8 | **无架构决策记录 (ADR)** | 技术选型（为什么用 SQLite 而不是 PostgreSQL？为什么 Vanilla JS 而不是 React？）无记录，未来重构时缺乏上下文。 | 在 docs/adr/ 下创建 ADR 模板，对关键决策记录背景、选项、结论、后果 |
